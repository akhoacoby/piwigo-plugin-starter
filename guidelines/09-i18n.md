# Internationalization (languages)

ALL user-facing strings go through `l10n` / `@translate`. Ship **en_UK AND fr_FR**.

## Rules
- PHP: `l10n('Human readable key')`. Templates: `{'Human readable key'|@translate}`.
- Use **human-readable English keys** (`$lang['Side by side']`), not opaque ids.
- Files: `language/<code>/plugin.lang.php` → `$lang['key'] = 'translation';`.
- REQUIRED locales: `en_UK` (source) + `fr_FR`. Keep keys in sync across both.
- Load once in `init()`: `load_language('plugin.lang', YOUR_PLUGIN_PATH);`.
- Missing key → `l10n` returns the key itself (English still renders), so never hardcode visible English outside `$lang`.
- Plurals: `l10n_dec($singular_key, $plural_key, $n)`. Args: `l10n('%d items', $n)`.
- ⚠️ `@translate` output is NOT HTML-escaped — escape if the string can contain data (see `05-frontend.md`).

## Adding a string
1. Add the key to `language/en_UK/plugin.lang.php` **and** `language/fr_FR/plugin.lang.php`.
2. Use it via `l10n()` / `{'…'|@translate}`.
3. Never leave a French key missing — both files must define every key.
