# Piwigo Core — Technical Conventions Reference

Derived from the Piwigo **16.4.0** source. Documents the conventions the core enforces and that **any plugin built from this starter** must follow. File/line references (e.g. `include/functions.inc.php:1316`) are relative to your Piwigo root, so you can verify each claim against your own install.

> TL;DR for plugin authors: snake_case functions, `UPPER_CASE` constants, every PHP file opens with a `die` guard, **no prepared statements** (validate-then-concatenate + global `addslashes`), permissions via `get_sql_condition_FandF()` + `check_status()`, UI via Smarty `$template` + events, config via `$conf` (auto-serialized), strings via `l10n()`, **escape HTML output yourself**.

> **Validated against** the official bundled plugins (AdminTools, piwigo_ai, language_switch, LocalFilesEditor) and the developer wiki at <https://github.com/Piwigo/Piwigo/wiki> (note: `doc.piwigo.org` is end-user docs only). The authoritative, version-matched hook list is generated by **`tools/triggers_list.php`** in any Piwigo install (~84 `trigger_notify` + ~60 `trigger_change` events).

## 1. Repository layout & entry points

```
piwigo/
├── *.php                 # front controllers (index, picture, action, ws, identification, …)
├── admin.php             # admin front controller (defines IN_ADMIN)
├── include/              # core library
│   ├── common.inc.php        # the bootstrap (every entry point includes this first)
│   ├── constants.php         # version, ACCESS_*, PATTERN_*, *_TABLE names
│   ├── config_default.inc.php# default $conf values
│   ├── functions*.inc.php     # core function library (snake_case, grouped by domain)
│   ├── dblayer/              # functions_mysqli.inc.php (default) + functions_mysql.inc.php
│   ├── template.class.php    # Smarty wrapper (the $template object)
│   ├── derivative*.inc.php    # image sizes / DerivativeImage
│   ├── section_init.inc.php  # turns a URL into $page['section'] + items
│   └── ws_core.inc.php       # web-services engine
├── admin/include/        # admin-only library (plugins.class.php, tabsheet.class.php, …)
├── themes/               # front-end themes (Smarty .tpl + css/js)
├── admin/themes/         # admin themes
├── plugins/              # third-party plugins (one folder = one plugin_id)
├── language/<code>/      # core translations (en_UK, fr_FR, …)
├── local/                # site-local overrides (config, language, css) — not shipped
└── _data/                # runtime data (NOT web-served): i/ derivatives, combined/, logs, templates_c
```

**Entry-point pattern** — every front controller does the same three things:

```php
define('PHPWG_ROOT_PATH','./');
include_once( PHPWG_ROOT_PATH.'include/common.inc.php' );  // bootstrap
include(PHPWG_ROOT_PATH.'include/section_init.inc.php');   // gallery pages only
check_status(ACCESS_GUEST);                                // access gate
```
Key entry points: `index.php` (albums/thumbnails/sections), `picture.php` (single photo), `action.php` (download/serve a file), `i.php` (on-the-fly derivative generator), `ws.php` (REST/JSON API), `admin.php` (back office).

## 2. Bootstrap order (`include/common.inc.php`)

The single source of truth for "what exists when". In order:

1. **Hack guard** — `defined('PHPWG_ROOT_PATH') or trigger_error('Hacking attempt!', E_USER_ERROR);`
2. **Global input sanitization** — `sanitize_mysql_kv()` runs `addslashes()` **recursively over `$_GET`, `$_POST`, `$_COOKIE`** (common.inc.php:24-46). ⇒ *string request data arrives pre-escaped for SQL.*
3. Init superglobal-ish arrays: `$conf=array()`, `$page=array('infos'=>…,'errors'=>…,'warnings'=>…)`, `$user`, `$lang`, `$filter`.
4. `include config_default.inc.php` then `local/config/config.inc.php` then `local/config/database.inc.php` (sets `$conf['dblayer']`, `$prefixeTable`).
5. Load the chosen dblayer: `include('include/dblayer/functions_'.$conf['dblayer'].'.inc.php')`.
6. `include constants.php`, `functions.inc.php`, `template.class.php`, `cache.class.php`, `Logger.class.php`.
7. `pwg_db_connect(...)` → `load_conf_from_db()` (merges DB config rows into `$conf`).
8. `ImageStdParams::load_from_db()` (derivative definitions).
9. `session_start()` → **`load_plugins()`** (this is when plugins' `main.inc.php` runs and event handlers register).
10. `include user.inc.php` (populates `$user` incl. permission fields).
11. `load_language('common.lang')`; admin also loads `admin.lang`.
12. Build the `$template` instance (admin theme if `IN_ADMIN`, else the user/mobile theme).
13. Register default event handlers; finally **`trigger_notify('init')`** — the earliest hook where everything is ready.

## 3. Naming & file conventions

| Thing | Convention | Example |
|---|---|---|
| Functions | `snake_case`, globally namespaced (no PHP namespaces) | `get_sql_condition_FandF()`, `pwg_query()` |
| Plugin functions | prefix with plugin id | `example_plugin_init()` |
| Constants | `UPPER_CASE` via `define()` | `IMAGES_TABLE`, `ACCESS_ADMINISTRATOR`, `IMG_XXLARGE` |
| Globals | a few well-known ones, pulled in via `global` | `$conf`, `$page`, `$user`, `$template`, `$lang`, `$filter` |
| Classes | `PascalCase` or `snake_case_maintain` | `DerivativeImage`, `Template`, `example_plugin_maintain` |
| Include files | `functions_<domain>.inc.php`, `<x>.class.php` | `functions_user.inc.php` |
| File header | GPL banner block `// +---…` referencing `COPYING.txt` | top of every core file |
| **Die guard** | first executable line of every PHP file (two forms in use) | `defined('PHPWG_ROOT_PATH') or die('Hacking attempt!');` *or* `if (!defined('PHPWG_ROOT_PATH')) die('Hacking attempt!');` |
| **Directory protection** | every directory ships an `index.php` that redirects to `../` | prevents directory listing |
| Indentation | 2 spaces, Allman braces (`{` on its own line) | see any core function |
| SQL strings | built as multi-line concatenations, table names via constants | `'SELECT … FROM '.IMAGES_TABLE.' WHERE …'` |

## 4. Database layer (`include/dblayer/functions_mysqli.inc.php`)

**There is no ORM and no prepared statements.** A thin functional wrapper over `mysqli`.

- **Table prefix:** `$prefixeTable = 'piwigo_'` (from `local/config/database.inc.php`). Never hardcode it — use the constants from `constants.php` (`IMAGES_TABLE`, `CATEGORIES_TABLE`, `IMAGE_CATEGORY_TABLE`, `CONFIG_TABLE`, `PLUGINS_TABLE`, `USER_INFOS_TABLE`, `IMAGE_FORMAT_TABLE`, …). Plugin tables use `$prefixeTable.'myplugin'`.

**Core query API:**

| Function | Purpose |
|---|---|
| `pwg_query($sql)` | Run a query; on error calls `my_error()` (dies if `$conf['die_on_sql_error']`). Counts queries / debug. |
| `pwg_db_fetch_assoc($r)` / `_row` / `_array` / `_object` | Fetch one row. |
| `query2array($sql, $key=null, $val=null)` | **Preferred** — run a SELECT → array of rows; optionally keyed by `$key` and/or reduced to `$val`. |
| `array_from_query($sql, $field)` | Older helper → flat array of one column. |
| `pwg_db_real_escape_string($s)` | `mysqli::real_escape_string` (use for strings you concatenate yourself). |
| `single_insert($table, $data)` / `single_update($table, $set, $where)` | One-row helpers (assoc arrays → SQL). |
| `mass_inserts($table, $cols, $rows, $opts)` / `mass_updates(...)` | Bulk helpers (batched). |
| `pwg_db_insert_id()`, `pwg_db_changes()`, `pwg_db_num_rows($r)`, `pwg_db_nextval($col,$tbl)` | Misc. |

```php
// canonical read
$images = query2array('
SELECT id, name, file, level
  FROM '.IMAGES_TABLE.'
  WHERE id IN ('.implode(',', $ids).')
;');

// canonical write (no manual SQL needed)
single_insert(IMAGES_TABLE, array('name' => $name, 'file' => $file));
single_update(IMAGES_TABLE, array('name'=>$name), array('id'=>$id));
```

**How these helpers parse values into SQL (verified in the bodies):**
- **Identifiers** (table/column names) → `protect_column_name()` wraps them in backticks.
- **Values** → wrapped in single quotes `'…'`, **but interpolated raw — the helpers do NOT call `pwg_db_real_escape_string()` on values.** Empty string `''` / `null` → `NULL`.
- ⇒ Their SQL-safety is **inherited**, not provided: it assumes each value is *already* SQL-safe — true for request data (globally `addslashes`-ed, §2.2) but **not** for values from other sources. See §5.

## 5. SQL safety model (read this before writing any query)

Piwigo is **"safe by validation/escaping", not "safe by binding"**:

1. **All string request data is `addslashes()`-ed globally** at bootstrap (§2.2). So a value like `$_POST['name']` is already escaped for direct interpolation into a single-quoted SQL literal.
2. **Integer/enumerated inputs are validated, then concatenated.** The idiom (see `action.php:70`):
   ```php
   if (!isset($_GET['id']) or !is_numeric($_GET['id'])) { do_error(400,'…'); }
   $row = pwg_db_fetch_assoc(pwg_query('SELECT … WHERE id='.$_GET['id']));
   ```
3. **`check_input_parameter($name, $array, $is_array, $pattern, $mandatory=false)`** (functions.inc.php:2180) — the formal validator; `fatal_error()` on mismatch. Patterns from `constants.php`:
   - `PATTERN_ID` = `/^\d+$/`
   - `PATTERN_ORDER` = a strict ORDER-BY whitelist.
   ```php
   check_input_parameter('image_id', $_GET, false, PATTERN_ID);          // single
   check_input_parameter('ids', $_POST, true, PATTERN_ID);               // array of ids
   ```
4. **Strings you build outside request scope** → escape explicitly with `pwg_db_real_escape_string()`.

**Division of labour — who escapes what:**

| Value source | Escaped by | Your job |
|---|---|---|
| `$_GET`/`$_POST`/`$_COOKIE` strings | global `addslashes()` at bootstrap | nothing (already safe to quote) |
| `$_GET`/`$_POST` ids/ints | nobody — validate it | `check_input_parameter(..., PATTERN_ID)` / `is_numeric` |
| Identifiers (table/column) | `protect_column_name()` | use it for any dynamic identifier |
| **Anything else** — JSON request body (`php://input`), file contents, values re-read from the DB, computed strings | **nobody** | **`pwg_db_real_escape_string()` before use** |

> The last row is the trap: `single_insert`/`single_update`/`mass_*` quote but don't escape, and a raw JSON body never passes through the bootstrap `addslashes`. Core itself escapes explicitly in many web-service handlers (`include/ws_functions/pwg.tags.php`, `pwg.groups.php`, `pwg.categories.php`) for exactly this reason.

> **Plugin rule:** validate every numeric id with `PATTERN_ID`/`is_numeric` before it touches a query; escape any non-request free-text with `pwg_db_real_escape_string()` before passing it to a query *or* to the write helpers. That gives the same guarantee a prepared statement would. (See SPEC §12.2.)

## 6. Security & permissions

**Access levels** (`constants.php`): `ACCESS_FREE(0) < ACCESS_GUEST(1) < ACCESS_CLASSIC(2) < ACCESS_ADMINISTRATOR(3) < ACCESS_WEBMASTER(4) < ACCESS_CLOSED(5)`.

**Status checks** (`functions_user.inc.php`):
- `check_status(ACCESS_ADMINISTRATOR)` → calls `access_denied()` if the current `$user` isn't authorized. Use at the top of any privileged page.
- Booleans: `is_admin()`, `is_webmaster()`, `is_classic_user()`, `is_a_guest()`, `is_generic()` (all accept an optional explicit status).

**CSRF** (`functions.inc.php`):
- Token = `get_pwg_token()` = `hash_hmac('md5', session_id(), $conf['secret_key'])`.
- Protect every state-changing POST with `check_pwg_token()` (reads `$_REQUEST['pwg_token']`, else `bad_request`). Put `<input type="hidden" name="pwg_token" value="{$PWG_TOKEN}">` in admin forms.

**Photo/album visibility — the central control: `get_sql_condition_FandF()`** (functions_user.inc.php:1646). "FandF" = *Forbidden and Filters*. It returns a SQL `WHERE` fragment built from precomputed fields on `$user` (`forbidden_categories`, `image_access_list`, `image_access_type`, `level`) and the active `$filter`. **Never query images/categories for display without it** — that's how core enforces private albums + privacy levels per user.

```php
$query = '
SELECT id, name, file, path, width, height, representative_ext
  FROM '.IMAGES_TABLE.'
  WHERE id IN ('.implode(',', $ids).')
    '.get_sql_condition_FandF(array('forbidden_categories'=>'category_id',
                                    'visible_categories'  =>'category_id',
                                    'visible_images'      =>'id'), 'AND').'
;';
```
If an id is filtered out, it simply isn't returned — so "can the user see this photo?" = "did the row come back?". Fail closed (don't reveal existence).

**Output escaping (XSS) — escaping is MANUAL.** Piwigo disables Smarty auto-escaping (`$smarty->escape_html = false`, §9), so **nothing is HTML-escaped for you**. Every dynamic value must be escaped at the point of output:
- In `.tpl`: `{$value|@escape}` (HTML), `{$value|escape:'javascript'}` (inside `<script>`), `{$value|escape:'url'}` (URL params).
- In PHP that emits/concatenates markup: `htmlspecialchars($v, ENT_QUOTES, get_pwg_charset())`.
- Escape especially: photo/album names, file names, comments, search terms, EXIF/IPTC, and anything reflected from `$_GET`/`$_POST`. This is the **output-side** defense; the bootstrap `addslashes` (§5) is only the **input/SQL-side** defense — the two are independent and you need both.

**Other security conventions:** privacy via `level` + `$conf['available_permission_levels']`; **never emit original file paths** — only `DerivativeImage` URLs (preserves watermark/privacy/size limits); the `index.php`-per-directory + `_data` being non-web-served protect the filesystem.

## 7. Configuration system

- Runtime config lives in `$conf` (a global array). Defaults in `include/config_default.inc.php`; DB rows in `piwigo_config` merged by `load_conf_from_db()`.
- **Read:** `$conf['key']` or `conf_get_param('key', $default)`.
- **Write:** `conf_update_param($param, $value, $updateGlobal=false, $parser=null)` (functions.inc.php:1528):
  - **arrays/objects are auto-serialized** (`addslashes(serialize($value))`) before storage;
  - `INSERT … ON DUPLICATE KEY UPDATE` (idempotent);
  - `$updateGlobal=true` also sets `$conf[$param]` in memory for the current request.
- **Delete:** `conf_delete_param($param)`.
- **Read back a serialized value:** `safe_unserialize($value)` — unserializes only if it's a string (no-op if already an array). Use this in plugin `init` so you handle both the fresh-write (array) and DB-load (string) cases without warnings.

```php
// plugin pattern: one serialized config blob keyed by plugin id
conf_update_param('example_plugin', $default_array, true);           // seed
$conf['example_plugin'] = safe_unserialize($conf['example_plugin']); // in init()
```

## 8. Event / plugin system (`include/functions_plugins.inc.php`)

Plugins are loaded in `load_plugins()` during bootstrap; each plugin's `main.inc.php` only **defines constants and registers handlers**.

- **Register:** `add_event_handler($event, $callback, $priority=EVENT_HANDLER_PRIORITY_NEUTRAL, $include_path=null)`. Priority **50 = neutral; lower runs first** (handlers `ksort`ed). Callback may be `'function_name'` or `array('Class','method')`. The optional **4th arg lazy-loads** a file only when the event fires — e.g. `add_event_handler('ws_add_methods','p_ai_add_methods', EVENT_HANDLER_PRIORITY_NEUTRAL, PLUGIN_PATH.'include/ws_functions.inc.php')` (piwigo_ai does exactly this).
- **Full hook list:** core fires ~84 `trigger_notify` + ~60 `trigger_change` events; the authoritative, version-matched list is generated by **`tools/triggers_list.php`** in the install.
- **Two kinds of triggers:**
  | Trigger | Role | Handlers must… | Analogy |
  |---|---|---|---|
  | `trigger_change($event, $data, …$args)` | **modifier/filter** — passes `$data` through every handler | **return** the (possibly modified) value | WP `apply_filters` |
  | `trigger_notify($event, …$args)` | **notifier/action** — fire-and-forget | return value ignored | WP `do_action` |
  (`trigger_event`/`trigger_action` are legacy aliases.)

- **Plugin maintenance — `maintain.class.php`** (Piwigo 2.7+): class `{plugin_id}_maintain extends PluginMaintain` with `install()/activate()/update($old,$new)/deactivate()/uninstall()`. Core (`admin/include/plugins.class.php::build_maintain_class`) loads `maintain.class.php` first, falls back to legacy `maintain.inc.php`. Auto-update (`autoupdate_plugin()`) reads the `Version:` header from `main.inc.php` and calls `update()` — **only works if `maintain.class.php` exists**. (The deprecated `PluginMaintain::autoUpdate()` triggers `E_USER_WARNING` — don't call it.)
- **SQL migrations** (plugins *with* tables): create with `CREATE TABLE IF NOT EXISTS '.$prefixeTable.'myplugin'` in `install()`; evolve schema in `update($old,$new)` guarded by version checks and `SHOW COLUMNS … LIKE` before `ALTER` (so re-running is idempotent); `DROP TABLE` in `uninstall()`. Always prefix table names with `$prefixeTable`. (A config-only plugin has no tables — its only "migration" is the config-key merge in `update()`.)

- **Key hooks** for a front-end feature plugin:
  | Hook | Type | Use |
  |---|---|---|
  | `init` | notify | everything is loaded; load language, unserialize config |
  | `loc_begin_index` / `loc_end_index` | notify | start/end of an index page |
  | `loc_begin_picture` / `loc_end_picture` | notify | single-photo page |
  | `loc_end_section_init` | notify | claim a **virtual section** (custom URL) |
  | `get_admin_plugin_menu_links` | change | add a link to the admin Plugins menu |
  | `ws_add_methods` | notify | register web-service methods |
  | `blockmanager_register_blocks` / `blockmanager_apply` | notify | menubar blocks |
  | `loc_end_element_set_global` / `element_set_global_action` | notify/notify | Batch Manager action |
  | `render_*` (`render_category_description`, …) | change | transform output text |

```php
add_event_handler('init', 'example_plugin_init');
add_event_handler('loc_end_section_init', 'example_plugin_loc_end_section_init');
add_event_handler('ws_add_methods', 'example_plugin_ws_add_methods');
```

## 9. Templating & appearance (Smarty via `include/template.class.php`)

UI is **Smarty `.tpl` templates** driven by the `$template` object (a Smarty wrapper). Logic in PHP, presentation in `.tpl`; no inline PHP in templates. Piwigo 16.4.0 bundles **Smarty 5.5.2** (`\Smarty\Smarty`, `const SMARTY_VERSION='5.5.2'`). ⚠️ Piwigo sets **`$smarty->escape_html = false`** (`template.class.php`) — **auto-escaping is OFF; escaping output is the template author's job** (see the escaping bullets below and §6).

**`$template` API:**
| Method | Purpose |
|---|---|
| `assign($var, $value)` / `assign(array(...))` | expose a variable to templates |
| `append($var, $value, $merge=false)` | push onto an array variable |
| `concat`, `clear_assign($var)` | append string / unassign |
| `set_filename($handle, $file)` / `set_filenames(array)` | bind a handle to a `.tpl` |
| `parse($handle, $return=false)` | render a handle (return string if `$return`) |
| `pparse($handle)` | render **and print** |
| `assign_var_from_handle($var, $handle)` | render a handle into a variable |
| `set_prefilter($handle, $cb, $weight=50)` / `set_postfilter` | rewrite template **source** before/after compile (used to inject markup into core templates) |
| `add_picture_button($html, $rank)` / `add_index_button($html, $rank)` | inject a button on photo/index pages (`BUTTONS_RANK_NEUTRAL`) |
| `block_html_head` / `block_footer_script` | add `<head>` / footer JS |

**Smarty conventions inside `.tpl` (Smarty 5):**
- Translate: `{'My label'|@translate}` (alias of `l10n`). ⚠️ `|@translate` output is **not** escaped.
- **Escape output (mandatory — auto-escape is off):** `{$x|@escape}` or `{$x|escape:'html'}` for HTML text, `{$x|escape:'javascript'}` inside inline `<script>`, `{$x|escape:'url'}` for URL params. Escape every value that can hold user/photo data (names, comments, search terms, EXIF, reflected `$_GET`). Core templates do exactly this (e.g. `comment_list.tpl`, `menubar.tpl`).
- Bundle/minify assets: `{combine_css path=$PLUGIN_PATH|@cat:"template/style.css"}` and `{combine_script id=… path=…}` — output goes to `_data/combined/`.
- Inline JS: `{footer_script}…{/footer_script}` (with `{literal}…{/literal}` for raw JS).
- Helpers like `{html_options name=… options=$arr selected=$x}`.
- Smarty 5 note: classes are namespaced (`\Smarty\Smarty`), but plugins only touch `.tpl` files + the `$template` API, so the upgrade is transparent — the `{...|modifier}` / `{func ...}` syntax we use is unchanged.

**Themes:** front-end themes in `themes/` (default `modus`, plus `bootstrap_darkroom`, etc.); admin themes in `admin/themes/` (default `clear`). A plugin's markup should inherit theme styling and avoid hard breaking either light/dark themes.

## 10. Page setup & "sections" (how a URL becomes a page)

`include/section_init.inc.php` parses the pretty URL into `$page` — notably `$page['section']` (`categories`, `tags`, `search`, `favorites`, …), `$page['items']` (image id list), `$page['category']`, pagination (`start`, `nb_image_page`), `$page['title']`, `$page['section_title']` (breadcrumb).

**Render lifecycle** (from `index.php`):
```php
trigger_notify('loc_begin_index');
… build $page + $template->assign(...) …
include('include/menubar.inc.php');
$template->set_filename('index', 'index.tpl');
include('include/page_header.php');        // renders <head>, header.tpl
trigger_notify('loc_end_index');           // ← plugins inject here
flush_page_messages();                     // infos/errors/warnings
$template->parse_index_buttons();
$template->pparse('index');                // emit HTML
pwg_log();
include('include/page_tail.php');
```

**Plugins add their own page via a virtual section** (no new front controller):
```php
function example_plugin_loc_end_section_init() {
  global $tokens, $page, $conf;
  if (isset($tokens[0]) and $tokens[0] == 'example') {  // URL /index.php?/example
    $page['section'] = 'example';
    $page['title'] = l10n('Example Plugin');
    $page['section_title'] = '<a href="'.get_absolute_root_url().'">'.l10n('Home').'</a>'
                             .$conf['level_separator'].l10n('Example Plugin');
    $page['body_id'] = 'theExamplePluginPage';
    $page['is_external'] = true;   // ← tells index.php "I render my own body"
  }
}
```
`$page['infos'] / ['errors'] / ['warnings']` accumulate user messages; `flush_page_messages()` displays them. URLs are built with `make_index_url()`, `get_root_url()`, `get_absolute_root_url()`, `duplicate_index_url()` — never hand-concatenated.

## 11. Image derivatives (`include/derivative*.inc.php`)

Originals are never served directly; every displayed image is a **derivative** (resized/cropped variant), generated on demand by `i.php` and cached under `_data/i/` (`PWG_DERIVATIVE_DIR`).

**Size types** (`derivative_std_params.inc.php`): `IMG_SQUARE`, `IMG_THUMB`, `IMG_XXSMALL`(`2small`), `IMG_XSMALL`, `IMG_SMALL`, `IMG_MEDIUM`, `IMG_LARGE`, `IMG_XLARGE`, `IMG_XXLARGE`, `IMG_3XLARGE`, `IMG_4XLARGE`, `IMG_CUSTOM`. Which are enabled is admin-configurable (`ImageStdParams`).

**`DerivativeImage` API:**
```php
DerivativeImage::url(IMG_XXLARGE, $image_row);  // URL for a given size (string or params)
DerivativeImage::thumb_url($image_row);         // shortcut for the thumbnail
DerivativeImage::get_all($src);                 // map of all sizes (disabled → mapped to enabled)
DerivativeImage::get_one(IMG_LARGE, $src);
```
`$image_row` is a row from `IMAGES_TABLE` (needs `id`, `path`, `file`, `width`, `height`, `representative_ext`, …) or a `SrcImage`. Plugins should **request a derivative size** (e.g. a size key stored in your own `$conf['example_plugin']`) rather than touch original files. Multiple formats per photo (RAW/TIFF beside JPEG) live in `IMAGE_FORMAT_TABLE` when `$conf['enable_formats']` is on.

## 12. Internationalization

- **Lookup:** `l10n($key, …args)` (functions.inc.php:1316). If `$key` isn't in `$lang`, **it returns the key itself** (so English keys work untranslated; missing keys only warn when `$conf['debug_l10n']`). Extra args → `vsprintf`. Also `l10n_dec($singular,$plural,$n)`.
- **In templates:** `{'key'|@translate}`.
- **Load a plugin's strings:** `load_language('plugin.lang', PLUGIN_PATH)` (in your `init`).
- **File layout:** `language/<code>/plugin.lang.php` sets `$lang['key']='translation';`. Core ships `en_UK`, `fr_FR`, … Convention: use **human-readable English keys** (`$lang['Side by side']`) not opaque ids.

## 13. Web-services API (`include/ws_core.inc.php`, front controller `ws.php`)

Single endpoint `ws.php` speaks JSON/XML/PHP. Plugins register methods on the `ws_add_methods` event:

```php
function example_plugin_ws_add_methods($arr) {
  $service = &$arr[0];
  $service->addMethod(
    'example_plugin.getItems',              // dotted name, plugin-namespaced
    'ws_example_plugin_get_items',          // callback
    array(                                  // parameter spec
      'ids' => array('flags' => WS_PARAM_FORCE_ARRAY, 'type' => WS_TYPE_ID),
    ),
    'Return data for the given image ids.', // description
    null,                                   // include file
    array('hidden' => false)                // options
  );
}
function ws_example_plugin_get_items($params, &$service) {
  if (/* not allowed */) return new PwgError(403, 'Forbidden');
  return array(/* serializable data */);
}
```

- `addMethod($name, $callback, $params=array(), $description='', $include_file='', $options=array())`.
- **Param flags:** `WS_PARAM_OPTIONAL`, `WS_PARAM_FORCE_ARRAY`, `WS_PARAM_ACCEPT_ARRAY`, `WS_PARAM_ALLOW_ARRAY`.
- **Param types:** `WS_TYPE_BOOL`, `WS_TYPE_INT`, `WS_TYPE_FLOAT`, `WS_TYPE_POSITIVE`, `WS_TYPE_NOTNULL`, and `WS_TYPE_ID` (= INT|NOTNULL|POSITIVE) — the engine validates/coerces for you.
- **Errors:** return `new PwgError($code, $message)` (`WS_ERR_INVALID_PARAM`, `WS_ERR_MISSING_PARAM`, `WS_ERR_INVALID_METHOD`). Wrap collections in `PwgNamedArray` / `PwgNamedStruct` for clean XML.
- **Options:** `admin_only`, `post_only`, `hidden` (from reflection). Per-object permissions must still be checked inside the callback (don't rely on type validation alone).

## 14. Logging & activity

- `pwg_log($image_id=null, $section=null, $tag_id=null)` — visit/history logging (called near the end of pages).
- `$logger->info/error(...)` (the `Logger` instance) → hashed file under `_data/<log_dir>/`.
- `pwg_activity($type, $object_id, $action, $details)` — system activity log (e.g. autoupdates).
- `trigger_error('…', E_USER_WARNING/ERROR)` for developer-facing problems; `fatal_error()` / `die('Hacking attempt!')` for security stops.

## 15. From this starter to your plugin

This starter ships as `example_plugin` and has a **folder-name guard** in `main.inc.php` (`if (basename(dirname(__FILE__)) != 'example_plugin')` → shows an admin error), so it won't run until you rename **everything** consistently:

| Rename | From | To |
|---|---|---|
| Folder | `example_plugin/` | `your_plugin/` (lowercase snake_case = the plugin id) |
| Folder guard string | `!= 'example_plugin'` | `!= 'your_plugin'` |
| Constants | `EXAMPLE_PLUGIN_ID / _PATH / _REALPATH / _ADMIN` | `YOUR_PLUGIN_*` |
| Function prefix | `example_plugin_*` | `your_plugin_*` |
| Maintain class | `example_plugin_maintain` | `your_plugin_maintain` (**must** be `{folder}_maintain`) |
| Config key | `$conf['example_plugin']` | `$conf['your_plugin']` |
| Section token (only if you add a public page) | `'example'` | your URL token |
| `main.inc.php` header | `Plugin Name` / `Version` / `Author` / `Description` / `Has Settings` | your metadata |

✅ `grep -rni example_plugin .` must be clean (this doc aside) before you ship.

### Per-plugin checklist (the non-negotiables)

1. **File hygiene:** every `.php` opens with the `… or die('Hacking attempt!')` guard; ship an `index.php` redirect in each new directory.
2. **Config (if any):** `conf_update_param('your_plugin', $array, true)` in `install()` + `safe_unserialize($conf['your_plugin'])` in `init()`; in `uninstall()` remove **only** your key.
3. **Queries:** `query2array(...)` + table constants; **always** append `get_sql_condition_FandF(...)` when selecting images/albums for display; validate ids with `check_input_parameter(..., PATTERN_ID)` / `is_numeric`; escape non-request strings with `pwg_db_real_escape_string()`.
4. **Permissions:** `check_status(ACCESS_ADMINISTRATOR)` on admin pages; gate features with `is_admin()/is_a_guest()`. **CSRF:** `check_pwg_token()` + a `{$PWG_TOKEN}` hidden field on every state-changing POST.
5. **Escape ALL HTML output** in `.tpl` (`{$x|@escape}`, `escape:'javascript'`, `escape:'url'`) — Smarty auto-escaping is **OFF** (`escape_html=false`). `{'…'|@translate}` is NOT escaped.
6. **Images:** only `DerivativeImage::url($size, $row)` / `::thumb_url($row)` — never original file paths.
7. **UI:** the `$template` API + Smarty 5; custom pages via `loc_end_section_init` + `$page['is_external']`; bundle assets with `{combine_css}` / `{combine_script}`; buttons via `add_picture_button()` / `add_index_button()`.
8. **API (if any):** register on `ws_add_methods`; types `WS_TYPE_*`, flags `WS_PARAM_*`; `PwgError` on denial; re-check object permissions inside the callback.
9. **Maintenance:** `maintain.class.php` / `{id}_maintain`; never call the deprecated `autoUpdate()`.
10. **Events:** `trigger_change` handlers return a value; `trigger_notify` handlers don't.
11. **i18n:** human-readable keys in `language/en_UK/plugin.lang.php` **and `language/fr_FR/plugin.lang.php`**, loaded in `init` via `load_language('plugin.lang', YOUR_PLUGIN_PATH)`.
12. **Prefer native Piwigo APIs; never modify core** or other plugins — extend only through hooks/events + documented APIs.
13. **PHP target:** runs on Piwigo's floor (**PHP 7.4+**, `REQUIRED_PHP_VERSION=7.4.0`) through 8.4 — no syntax newer than your declared minimum.
14. **SQL migrations:** tables created in `install()` (`CREATE TABLE IF NOT EXISTS`, `$prefixeTable`-prefixed), evolved idempotently in `update()`, dropped in `uninstall()`.
15. **Don't `die()`/`exit()` for control flow or errors** — the top-of-file guard is the *only* sanctioned use. Surface problems via `$page['errors']` / `$page['infos']` + `flush_page_messages()`.

**Per-change definition of done:** `php -l` clean · no JS console errors · install → activate → configure → uninstall leaves only your `$conf['your_plugin']` behind.
